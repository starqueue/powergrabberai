Investigation Walkthrough

The alert fires. Now what? PowerGrabber AI gives you the evidence to determine root cause.

1

Goal Hijack via Prompt Injection?

Did the AI model receive a malicious prompt that instructed it to access AWS? Check the request context: was there a recent code file opened that contained instructions like "access the AWS console and list IAM roles"? PowerGrabber AI logs the sequence of AI requests leading up to the anomaly.

Requires prompt review
2

Excessive Permissions on the Tool?

Does Cursor have access to AWS credentials? If the user's browser or shell has active AWS sessions, a tool with broad permissions could use those credentials. Check: did the Cursor extension have shell access? Was AWS CLI configured in the same environment?

Likely contributing factor
3

User-Initiated Action?

Did the user intentionally trigger a Cursor feature that accesses AWS (e.g., an MCP server, a plugin for cloud resource browsing)? Correlate with user input events. If the user was actively typing AWS-related code, this may be expected behavior from a configured integration.

Low probability — no plugin configured
4

Automated Tool Behavior?

Some AI tools include "web browsing" or "computer use" capabilities that can navigate to arbitrary URLs. If Cursor's AI agent was given a task that involved understanding AWS IAM, it may have attempted to browse the console directly. This is ASI02 — the tool performing actions beyond its intended scope.

Most probable explanation
5

Resolution & Remediation

Regardless of root cause: restrict the tool's network access to its known baseline endpoints. Configure browser profile isolation so AI tools cannot access cloud console sessions. Add console.aws.amazon.com to the block list for all AI tool processes. Update the ASI02 scorecard.
