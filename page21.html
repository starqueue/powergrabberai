Why Not a Proxy?

🔒

No Certificate Trust Store Modification

Proxy-based tools require installing a custom CA certificate in the OS or browser trust store. This weakens TLS for all traffic, not just AI calls. PowerGrabber never touches your certificates.

🛡

No Broken TLS / No MITM

Proxies decrypt and re-encrypt traffic (TLS interception). This breaks certificate pinning, confuses security tools, and creates a new attack surface. PowerGrabber reads request metadata at the browser layer before encryption.

🌐

No Network Topology Changes

No PAC files. No system proxy settings. No DNS changes. No firewall rules. The browser extension runs in-process. The WebSocket connects to localhost. Nothing leaves the machine differently.

⚡

Zero Performance Impact

The webRequest API is event-driven and non-blocking. No additional network hops. No TLS handshake overhead. No added latency. Requests flow directly to the API provider.

How It Works, Step by Step

Browser makes an AI API request

A user or AI tool in the browser sends a request to api.openai.com, api.anthropic.com, or any configured AI endpoint. The request has not left the browser yet.

webRequest API fires

The extension's onBeforeSendHeaders listener captures the URL, method, and headers (including Authorization, model parameters, and API keys). This happens at the browser layer, before TLS encryption.

Metadata sent via local WebSocket

The captured metadata is sent over a WebSocket connection to ws://127.0.0.1:9753. This never leaves the machine. No cloud. No external service.

Rust backend parses and stores

The backend identifies the AI provider, extracts model name, token counts, and rate-limit headers. It computes cost, attributes the call to a tool and team, and writes to local SQLite.

Response headers captured on completion

onCompleted fires when the response arrives. Rate-limit headers (x-ratelimit-remaining-tokens, x-ratelimit-remaining-requests) are captured and forwarded to the backend for quota tracking.

Request flows normally

The original request is never modified, delayed, or rerouted. TLS is intact. The API provider sees a normal request from the browser. Zero interference.
