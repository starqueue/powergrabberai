Policy Engine Specification

The AI Policy Engine

Define rules in TOML. PowerGrabber evaluates every captured AI request against your policy set in real time. Block, alert, log, or rate-limit — per provider, per tool, per schedule.

How It Works

Every event. Every rule. Every time.

When PowerGrabber intercepts an AI API request, it extracts metadata (provider, model, tool, user, timestamp, token count) and runs it through your complete policy rule set. Matching rules trigger actions: block, alert, log, or rate-limit.

Policy Rules

TOML Policy Syntax

Five real-world rules covering the most common governance scenarios. Copy these into your powergrabber.policy.toml and customise.

powergrabber.policy.toml TOML
# ────────────────────────────────────────── # Rule 1: Block unapproved AI providers # ────────────────────────────────────────── [[rule]] name = "block-unapproved-providers" description = "Only allow OpenAI and Anthropic. Block everything else." condition = "provider NOT IN ['openai', 'anthropic']" action = "block" severity = "critical" notify = ["slack:#security-alerts", "webhook:pagerduty"] # ────────────────────────────────────────── # Rule 2: Rate-limit alerts per tool # ────────────────────────────────────────── [[rule]] name = "rate-limit-per-tool" description = "Alert if any single tool exceeds 500 requests in 10 minutes." condition = "tool.request_count(window='10m') > 500" action = "alert" severity = "high" cooldown = "15m" notify = ["slack:#ai-ops"] # ────────────────────────────────────────── # Rule 3: Block AI requests from healthcare-origin apps # ────────────────────────────────────────── [[rule]] name = "healthcare-origin-block" description = "Block requests where the originating app handles PHI/ePHI." condition = "tool.tag CONTAINS 'healthcare' AND provider.region != 'us-hipaa'" action = "block" severity = "critical" log_body = false notify = ["slack:#compliance", "email:dpo@company.com"] # ────────────────────────────────────────── # Rule 4: After-hours detection # ────────────────────────────────────────── [[rule]] name = "after-hours-ai-usage" description = "Flag AI usage outside business hours (potential automation/exfil)." condition = "event.time.hour NOT IN range(8, 18) AND event.day NOT IN ['Sat','Sun']" action = "alert" severity = "medium" notify = ["log:after-hours.jsonl"] # ────────────────────────────────────────── # Rule 5: New tool detection # ────────────────────────────────────────── [[rule]] name = "new-tool-alert" description = "Alert when a previously unseen tool binary makes its first AI call." condition = "tool.binary NOT IN known_tools()" action = "alert" severity = "high" auto_add = false notify = ["slack:#security-alerts", "webhook:siem"]

block-unapproved-providers

Maintains an allowlist of sanctioned AI providers. Any request to an unlisted provider (Cohere, Mistral, local LLMs) is blocked at the proxy and flagged to security.

rate-limit-per-tool

Sliding-window counter per tool binary. Catches runaway loops, accidental infinite retries, or automated scripts consuming tokens at abnormal velocity.

healthcare-origin-block

Tags from your CMDB identify healthcare applications. If one sends data to a non-HIPAA AI endpoint, the request is hard-blocked and compliance is notified immediately.

after-hours-detection

AI usage at 2 AM on a Tuesday is either automation or something unexpected. This rule creates a forensic log without blocking, letting SecOps investigate at their pace.

new-tool-alert

Shadow AI starts with "just trying this new tool." This rule fires the moment an unknown binary makes its first API call, giving your team instant visibility into tool sprawl.
