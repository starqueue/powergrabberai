How It Works
Model IDs from URL Paths
PowerGrabber extracts model identifiers automatically from the structure of AI API requests. No body inspection required.
01

URL Path Parsing

AI APIs encode model IDs in predictable URL patterns. OpenAI uses /v1/chat/completions with model in the body, but the x-model response header reveals it. Anthropic's /v1/messages returns the model in response headers. Google embeds it directly: /v1/models/gemini-2.5-flash:generateContent.

02

Response Header Extraction

Most AI providers return the actual model version used in response headers or the response metadata visible in HTTP trailers. PowerGrabber captures these to distinguish between gpt-4o and gpt-4o-2024-11-20, giving you version-level visibility.

03

Inventory Aggregation

Each unique model ID is tracked over time: first seen date, last seen date, request count, originating tools (identified by User-Agent headers), and user count. New models trigger a "shadow model" alert if they are not on your approved list.

Governance Features
Beyond Discovery
The model inventory powers ongoing governance workflows.
!

Shadow Model Detection

Any model not on your approved list is flagged as a shadow model the moment it first appears in traffic. Get alerted before unapproved models become embedded in production workflows.

↻

Deprecated Model Flagging

PowerGrabber maintains a registry of deprecated model versions and their sunset dates. When traffic to a deprecated model is detected, the responsible team is notified with migration guidance.

⇅

Model Version Drift

Track when providers silently upgrade model versions behind stable aliases. If "gpt-4o" starts resolving to a new snapshot version, PowerGrabber detects the change and logs it.

☰

Tool-to-Model Mapping

See which internal tools and applications are calling which models. Know that Cursor uses GPT-4o, your Slack bot uses Haiku, and an unknown script is still hitting a deprecated endpoint.
