Why Shared Keys Break Accountability

🔎 No Attribution

When four tools share one key, the OpenAI usage dashboard shows one line item. A $400 spike could be Cursor autocomplete, a runaway script, or an employee's side project. You cannot attribute cost to the tool that caused it.

🔒 No Revocation Granularity

If one tool is compromised, revoking the key kills all four tools. There is no way to surgically disable the bad actor without collateral damage to every team using the same credential.

📈 No Rate Control

One tool hitting rate limits means every tool gets throttled. Shared keys create shared blast radius. A poorly-written script can take down your entire AI-assisted development workflow.

📑 No Audit Trail

SOC 2 and ISO 27001 require per-system access logs. A shared key means your audit trail says "OpenAI was accessed 34,000 times" but cannot say which application made each request.

How PowerGrabber Detects This — Without Storing Keys

01

Prefix Extraction

The proxy reads only the first 10 characters of each Authorization header value. The full key is never logged, stored, or written to disk. Only the prefix is retained for matching.

02

Org ID from Headers

OpenAI requests include an OpenAI-Organization header. PowerGrabber extracts the org ID to correlate which organisation the key belongs to, without needing the key itself.

03

Tool Fingerprinting

Each request is tagged with the originating process (PID, binary name, command line). When the same key prefix appears across multiple tool fingerprints, a cross-tool reuse event is emitted.

04

Policy Evaluation

TOML policy rules can trigger on credential.reuse_count > 1. Alerts fire via webhook, Slack, or local log, giving your IAM team immediate visibility.
